# 学习协作方式

这个仓库默认采用一种适合“有编程经验，但在学新语言或新技术”的协作方式：

- 我负责讲解、拆解和引导。
- 你负责手动抄写、运行、观察和提问。

目标不是直接拿到现成答案，而是通过少量示例和亲手输入，真正建立语感和理解。

## 适用范围

这套方式适用于当前仓库中的所有语言或技术目录，例如：

- `java`
- `go`
- `python`
- `rust`
- `typescript`
- `haskell`

以后新增其他语言目录，也可以继续沿用。

## 协作分工

### 我负责

- 根据当前阶段给出少量示例代码。
- 解释语法、思路和常见用法。
- 控制代码长度，保证适合手抄。
- 出练习题、改错题和思考题。
- 根据你的运行结果继续推进下一步。
- 在你卡住时帮你定位报错、解释原因、给修正方向。

### 你负责

- 手动把代码抄写到本地文件。
- 自己运行代码并观察结果。
- 尝试先独立理解，再带着具体问题来问。
- 把输出结果、报错信息或不理解的点发给我。
- 尽量自己完成练习，不直接跳到答案。

## 代码提供原则

- 默认不给整套现成项目代码。
- 默认不给一大段完整答案。
- 优先给“当前知识点所需的最小示例”。
- 每次只推进一个小主题，避免一下子塞太多内容。
- 如果是练习，我只给方向、要求和必要提示，不直接替你写完。

## 推荐学习节奏

每一小节建议按这个顺序进行：

1. 我先讲一个小知识点。
2. 我给一小段示例代码。
3. 你手动抄写到对应文件。
4. 你自己运行并观察输出。
5. 你把结果、报错或疑问发我。
6. 我继续解释并推进下一小节。
7. 我布置一个小练习让你自己补。

## 遇到报错时的处理方式

如果运行失败，优先按这个方式协作：

1. 不急着重写整段代码。
2. 把报错原文发给我。
3. 如果有输出，也一起发我。
4. 如果你怀疑是自己抄错了，也可以把相关代码贴出来。

我会优先帮你：

- 判断是语法错误、类型错误还是运行时错误。
- 解释为什么错。
- 给你修改方向。
- 尽量不直接把完整正确答案贴给你。

## 文件组织建议

每个语言目录都可以按下面方式组织：

- 学习计划文档
- 分阶段目录
- 对应主题的练习文件
- 尽量保留“课程骨架”，不要默认塞满完整答案

如果某个目录采用这套方式，文件里更适合放：

- 本课目标
- 你需要自己补的内容
- 练习题
- 思考题

而不是直接放完整实现。

## 什么时候可以给完整代码

下面这些场景可以适当放宽：

- 你明确说“这次我想先看完整版本”。
- 你已经自己写过一遍，现在要对照参考答案。
- 当前目标是做项目，而不是训练基础语感。
- 某个错误太琐碎，继续手猜反而降低学习效率。

但默认仍然优先采用“少量示例 + 手抄 + 反馈”的节奏。

## 一个通用开场模板

以后学习任意新语言时，都可以默认按这个方式开始：

1. 先确定学习目标。
2. 我给出第一个最小知识点。
3. 你手动抄写并运行。
4. 我根据结果继续讲。

例如：

- 学 Java：先从类、方法、对象开始。
- 学 Go：先从变量、函数、错误处理开始。
- 学 TypeScript：先从 JavaScript 核心和类型标注开始。
- 学 Rust：先从所有权、变量和函数开始。

## 核心原则

- 以理解为主，不以复制为主。
- 以动手为主，不以浏览为主。
- 以小步推进为主，不以一次学完为主。
- 以反馈驱动为主，不以单向灌输为主。
